@Sites

Build, test, and publish a polished website for our hackathon project.

## Project Brief

Project name:
[ENTER A SHORT, MEMORABLE NAME]

Problem:
[WHAT PROBLEM ARE YOU TRYING TO SOLVE?]

Proposed solution:
[DESCRIBE YOUR SOLUTION IN 2–4 SENTENCES]

Target users:
[WHO WILL USE OR BENEFIT FROM THIS?]

Core demo:
[DESCRIBE THE ONE IMPORTANT THING A VISITOR SHOULD BE ABLE TO DO]

Content, data, and assets:
[PASTE RELEVANT CONTENT HERE AND/OR ATTACH FILES, IMAGES, DATA, OR LINKS]

## Goal

Create a focused, functional prototype that a team can demonstrate in no more than two minutes.

The site should immediately help visitors understand:

1. The problem
2. The proposed solution
3. Who benefits
4. How the solution works
5. Why the idea matters

Start building immediately. Make sensible assumptions and proceed without asking us to make routine design or technical decisions.

## Website Requirements

- Create a clear, engaging landing section with the project name, value proposition, and a prominent call to action.
- Implement one complete and functional user journey based on the “Core demo” above.
- Prioritise a working prototype over a large number of incomplete features.
- Make all buttons, links, forms, filters, navigation elements, and interactive components functional.
- Do not include fake buttons, dead links, or controls that appear functional but do nothing.
- Use realistic sample data when real data is unavailable. Clearly label it as demonstration data.
- Avoid external APIs, authentication, databases, or complicated setup unless they are essential to the idea.
- When a real integration is unavailable, create a convincing simulated version that can be demonstrated.
- Make the site responsive and usable on desktop and mobile.
- Use a modern, visually coherent design with readable typography, strong contrast, clear hierarchy, and accessible controls.
- Include appropriate success, loading, empty, and error states where relevant.
- Do not include confidential, sensitive, or personally identifiable information.

## Suggested Page Structure

Use the following structure where appropriate:

1. Hero section — project name, value proposition, and primary action
2. Problem — concise explanation of the challenge
3. Solution — how the proposed concept addresses it
4. Interactive demonstration — the main functional experience
5. Benefits or impact — why the solution is valuable
6. How it works — simple three-step explanation
7. Team section — optional team name and members
8. Final call to action

You may change this structure when another layout would better communicate the idea.

## Quality Check

Before publishing:

1. Preview and run the website.
2. Test the complete main user journey.
3. Test the site at desktop and mobile widths.
4. Fix broken interactions, build errors, console errors, spelling mistakes, and layout problems.
5. Remove unnecessary sections and placeholder content.
6. Ensure the project can be understood without an oral explanation.
7. Ensure the most impressive feature is easy to find and demonstrate.

## Publishing

After completing and validating the website:

1. Save the final version.
2. Deploy it through Sites hosting.
3. Set access to “Anyone on the Internet” when public publishing is available. Otherwise, use the broadest sharing option permitted by the workspace.
4. Return the live Site URL.

Do not stop after merely generating source code. Continue through previewing, testing, correcting, saving, and publishing the finished Site.

If deployment is blocked by account permissions or workspace policy, identify the exact blocked step and leave the fully tested Site ready for an organiser to publish.
